# 第二步：空间定位与数据化呈现

第二步，推进空间定位与数据化呈现，搭建单体诊断与空间地图的融合基础。在完成单体诊断功能开发后，项目进一步引入空间地图能力，并依托微信小程序与云开发完成障碍点信息的采集、存储、检索与可视化展示。结合当前代码实现，具体流程可概括为以下四个环节：

① 小程序端采集与辅助分析。用户在问题上报页面拍摄或选择现场图片，填写问题描述，并选择问题类别与所属社区；页面同时预留“生成报告”入口，可结合图片与文本生成 `aiSolution` 作为辅助分析内容。位置信息采集以 `wx.chooseLocation` 手动选点为主，在位置选择失败时回退使用 `wx.getLocation` 获取当前坐标。前端最终将图片、经纬度、问题类别、问题描述、社区名称、用户建议及辅助分析结果等字段绑定为统一上传对象。

② 数据上传与云存储。小程序首先通过 `wx.cloud.uploadFile` 将现场图片上传至云存储，获得对应的 `fileID` 列表；随后再通过 `wx.cloud.callFunction` 调用 `createIssuePost` 云函数，将位置坐标、图片标识与文本字段统一写入云开发数据库。云函数内部会将前端传入的经纬度转换为云数据库支持的 `GeoPoint` 类型，并同步保存问题类别、地址信息、用户建议与辅助分析内容等结构化数据。

③ 数据库设计。当前项目的真实实现并非 `PostgreSQL + PostGIS`，而是基于微信云开发 `CloudBase` 的文档型数据库进行存储。以 `posts` 集合中的问题上报数据为例，每条记录主要包含：`_id`（唯一标识）、`_openid`（提交用户标识）、`type`、`status`、`title`、`content`（问题描述）、`images`（图片 `fileID` 列表）、`category/categoryId/categoryName`（问题类别）、`community`（社区名称）、`location`（`GeoPoint` 空间坐标）、`address/formattedAddress/detailAddress`（地址信息）、`userSuggestion`（用户建议）、`aiSolution`（辅助分析结果）、`stats`（互动统计）以及 `createTime/updateTime`（时间戳）等字段，从而形成面向障碍点上报与后续分析的数据底座。

④ 空间检索与可视化。地图首页通过云函数 `getPublicData` 传入中心点与检索半径，并在后端使用 `geoNear` 实现附近问题帖的快速检索；前端在获得问题帖子后，进一步提取 `GeoPoint` 中的经纬度信息，构建 `markers` 数组并交由微信地图组件渲染，实现社区障碍点在地图上的分布展示。用户点击点位后可继续进入帖子详情查看问题描述、位置和后续互动信息；在路线分析场景下，系统还会对靠近规划路线的障碍点进行额外标记，从而为社区级问题分布分析和后续诊断报告生成提供直观的空间数据支撑。
